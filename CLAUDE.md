# DeskBuddy

A desk companion device for a WeWork office. A 1.8" AMOLED screen shows an animated pixel-art plant-ghost that reflects plant health (soil moisture, light) and work context (location via WiFi BSSID, Jira task, Google Calendar meetings).

## Architecture

Three devices on the same WiFi:

- **MacBook Air M2** — runs a BSSID location daemon (LaunchAgent, every 30s). Reads WiFi access point, maps to office location, POSTs to Mac Mini.
- **Mac Mini** (always on at office) — Node.js + Express server at `deskbuddy.local:3777`. Polls Google Calendar (60s) and Jira (60s), receives location + sensor data, runs a 10-state priority cascade engine, serves `GET /status` to the ESP32.
- **Waveshare ESP32-S3-Touch-AMOLED-1.8** — the desk device. Reads moisture + light every 10s, POSTs to Mac Mini. GETs `/status` every 30s, renders sprite + alternating text + edge glow via LVGL.

## Hardware

Only 3 items to buy (~₹3,000 total):
1. **Waveshare ESP32-S3-Touch-AMOLED-1.8** — all-in-one board with 368×448 AMOLED, touch, mic, speaker, IMU, RTC, case
2. **Capacitive Soil Moisture Sensor v2.0** — analog, on GPIO17 (ADC2 ch6)
3. **GY-302 BH1750 Light Sensor** — I2C at 0x23, on GPIO14 (SCL) / GPIO15 (SDA)

## Key Technical Decisions

- **No external HTTP deps** — Calendar and Jira integrations use Node's native `https` module. Only deps are `express` and `cors`.
- **ADC2 + WiFi works on ESP32-S3** — hardware arbiter (NOT the same as original ESP32 hard block). Retry logic in sensors.h handles occasional timeouts.
- **No BME280** — temperature was dropped from scope, not used in any state logic.
- **No Claude AI in v1** — rubber duck chat is documented as future enhancement.
- **No co-author tags** — never add "Generated by Claude", "AI-assisted", or similar attribution anywhere in code, comments, commits, or docs.

## State Engine (10 states, priority cascade)

Evaluated top-to-bottom, first match wins:

1. Plant emergency → `thirsty` (moisture <20) or `overwatered` (>90)
2. Manual override → whatever was set via POST /override
3. Touch sensor → `loved` (10s hold after tap)
4. Darkness → `sleeping` (light <50 lux)
5. User away → `sleeping` (no location update 5min)
6. Calendar → `meeting` (inMeeting = true)
7. Location → `rooftop` (roof) or `focus` (cubicle)
8. Jira → `coding` (in-progress task) or `idle` (none)
9. Fallback → `happy`

## API Endpoints

| Method | Path | Who calls it | Purpose |
|--------|------|-------------|---------|
| GET | /status | ESP32 every 30s | Resolved state + all data |
| POST | /sensors | ESP32 every 10s | Push moisture, light, touched |
| POST | /location | MacBook every 30s | Push BSSID location |
| POST | /override | Dashboard | Force a state (expires) |
| GET | /dashboard | Browser | Debug web UI |
| GET | /health | Any | Server health check |

## Display Layout

- Sprite fills ~80% of 368×448 AMOLED (48×48 base scaled ~7×)
- One text line at bottom alternating every 5s (location ↔ state)
- Edge glow: green = plant healthy, red = needs water, pulsing sine wave
- AMOLED true blacks = unused pixels are off

## Development Commands

```bash
# Server
cd server && npm install && cp .env.example .env && npm start
# or dev mode with auto-reload:
npm run dev

# Firmware (requires PlatformIO CLI)
cd firmware && pio run              # build
cd firmware && pio run -t upload    # flash to board
cd firmware && pio device monitor   # serial output

# Sprite conversion
python scripts/png_to_rgb565.py input.png --name happy --frames 4 --size 48
```

## Coding Conventions

- Server code: CommonJS (`require`), no semicolons optional (current code uses them — stay consistent)
- Firmware: C++ with Arduino framework, PlatformIO build system
- Keep server dependency-free beyond express/cors — use native `https`, native `fs`, etc.
- Sensor thresholds and timeouts are configurable via .env, with sensible defaults in code
- All state logic lives in state-engine.js — stores are dumb data holders, integrations are dumb pollers
